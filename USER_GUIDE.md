## 1.1 目标

OrderSphere‑Hub 是基于 Python + FastAPI + Vue3 + SQLite 技术栈构建的全栈 Web 在线点餐系统。本文档仅定义并覆盖该系统的核心业务子系统，包括：菜单展示、JSON 菜单自动导入、购物车、订单创建与查询、管理端扩展接口、开发环境一键启动、生产环境静态文件托管等功能模块。本文档不涉及系统以外的硬件设备、第三方支付、多门店连锁管理及其他未提及的子系统或外部组件。

# 2. 综合描述

`运行环境`： 本系统支持跨平台运行，具体环境要求如下：  
- 开发环境：Windows 10及以上、macOS 12及以上、Linux（Ubuntu 20.04及以上），需安装Python 3.8+、Node.js 16+，依赖FastAPI、Vue3（Vite）相关组件及SQLite 3.30+；  
- 生产环境：支持上述所有操作系统，可通过FastAPI直接托管前端静态文件，无需额外部署Web服务器，建议配置基础网络环境（支持HTTP访问）；  
- 客户端环境：兼容Chrome、Firefox、Edge等现代浏览器最新版本，支持响应式访问，无需安装额外客户端软件。  
`产品用户`： 本系统用户分为两类，均为系统核心使用者，具体如下：  
- 普通用户：餐厅顾客，核心需求为浏览菜单、添加商品至购物车、创建订单、查询个人订单信息，无需注册，可直接使用核心点餐功能；  
- 管理员：餐厅管理人员，使用默认账户（username: admin, password: admin123）登录，核心需求为通过管理端接口管理菜单、查看所有订单、维护系统基础配置。  
`已知限制`： 本系统仅聚焦核心线上点餐功能，存在以下明确限制，不涉及相关扩展能力：  
- 功能限制：不支持第三方支付集成（仅实现订单创建与查询，无实际支付流程）、不支持多门店连锁管理（仅适配单餐厅模式）、不集成线下硬件（如收银机、打印机等）；  
- 性能限制：因采用SQLite文件型数据库，不适合高并发场景，建议单餐厅同时在线用户不超过50人；  
- 扩展限制：系统架构仅支持基础功能迭代，不支持大规模定制化开发，管理端仅提供基础接口，无可视化管理界面。  
`假设`： 本系统的正常运行基于以下合理假设，若假设不成立，可能影响系统功能可用性：  
- 假设用户具备基础的浏览器操作能力，普通用户可独立完成浏览菜单、下单等操作，管理员具备基础的系统操作认知；  
- 假设运行环境网络稳定，无频繁断网、网络延迟过高的情况，确保前端与后端接口正常通信；  
- 假设JSON菜单导入文件格式符合系统预设规范，无格式错误、数据缺失等问题；  
- 假设管理员妥善保管账户信息，不泄露账号密码，避免非授权人员操作管理端接口。  
`依赖`： 本系统的正常开发、部署与运行，依赖以下组件、环境及规范：  
- 技术依赖：依赖Python 3.8+运行环境、Node.js 16+（用于前端构建）、FastAPI框架、Vue3（Vite）框架及SQLite 3.30+数据库；  
- 文件依赖：依赖符合规范的JSON菜单文件，用于系统初始化时自动导入菜单数据；  
- 环境依赖：依赖开发/生产环境的基础网络配置、权限配置，确保系统可正常启动、访问及数据持久化；  
- 操作依赖：依赖管理员按规范操作管理端接口，确保系统数据的准确性与安全性。

## 2.1 产品的前景

### 2.1.1 产品的背景与起源  
随着餐饮行业数字化转型加速，中小餐厅对线上点餐解决方案的需求日益迫切，传统人工点餐模式存在效率低下、易出错、人力成本高的痛点，而现有线上点餐系统多存在部署复杂、成本较高、功能冗余（适配多门店、多业态）等问题，难以满足中小单餐厅“轻量、便捷、低成本”的核心需求。

基于此，OrderSphere-Hub 在线餐厅点餐系统应运而生，其起源于对中小餐厅线上点餐场景的精准需求挖掘，旨在打造一款无需复杂配置、部署便捷、聚焦核心功能的全栈在线点餐产品，解决中小餐厅线上化转型的痛点，降低其数字化门槛，同时为顾客提供简洁、高效的线上点餐体验。

### 2.1.2 产品定位说明

## 2.4 运行环境

OrderSphere-Hub 在线餐厅点餐系统为全栈Web应用，基于 Python+FastAPI+Vue3+SQLite 技术栈开发，运行环境分为开发环境与生产环境，涵盖硬件平台、操作系统、软件组件等多个维度，具体要求如下，确保系统稳定、流畅运行。

`服务器端`：最低配置为CPU Intel Core i3 及以上、内存4GB 及以上、硬盘空间10GB 及以上（用于存储系统文件、SQLite数据库文件、前端静态文件）；推荐配置为CPU Intel Core i5 及以上、内存8GB 及以上、硬盘空间20GB 及以上，支持有线网络连接，保障网络稳定性。  
`客户端`：普通点餐用户、管理员、临时访客使用的终端设备（电脑、手机、平板）无特殊硬件要求，具备基础的浏览器运行能力、网络连接能力即可，手机/平板需支持响应式显示，确保页面正常渲染。

系统支持跨平台部署与访问，服务器端（部署环境）与客户端（访问环境）操作系统要求如下：  
-`服务器端`：支持Windows、macOS、Linux三大主流操作系统，具体版本要求：  
-  Windows：Windows 10 及以上（64位）；  
-   macOS：macOS 12 及以上；  
-   Linux：Ubuntu 20.04 及以上、CentOS 8 及以上（64位）。  
-`客户端`：无操作系统限制，只要终端设备安装兼容的浏览器即可，支持Windows、macOS、iOS、Android等主流操作系统的最新版本。

### 2.4.3 软件组件要求  
系统运行依赖特定的软件组件、框架及工具，需提前安装配置，具体要求如下：  
-`后端相关组件`：Python 3.8 及以上版本、FastAPI 0.100.0 及以上版本、Uvicorn 0.23.2 及以上版本（FastAPI运行服务器）、SQLite 3.30.0 及以上版本（数据库）；  
-`前段相关组件`：Node.js 16.0.0 及以上版本、Vue3（Vite 4.0.0 及以上版本）、Element Plus 2.0.0 及以上版本（UI组件库）；  
-`浏览器组件`：客户端访问需使用现代浏览器最新版本，支持Chrome 100.0 及以上、Firefox 98.0 及以上、Edge 100.0 及以上，确保页面交互、数据渲染正常；  
-`其它工具`：开发环境需安装Git（版本控制工具），用于代码管理；生产环境无需额外工具，FastAPI可直接托管前端静态文件。

### 2.4.4 共存应用程序要求  
系统运行过程中，可与以下应用程序共存，无冲突影响，具体如下：  
- 基础办公软件（如Office、WPS）：此类软件与系统无关联，共存时不影响系统运行；  
- 网络安全软件（如防火墙、杀毒软件）：需确保安全软件允许系统服务（FastAPI）、浏览器正常访问网络，避免拦截系统接口请求；  
- 其他轻量应用程序：如文本编辑器、终端工具等，共存时无资源冲突，不影响系统稳定性。

注：系统不建议与大型占用内存、占用网络带宽的应用程序（如大型游戏、视频剪辑软件）在同一服务器端共存，避免资源竞争导致系统运行卡顿、响应缓慢。

## 3.2 硬件接口

本节定义 OrderSphere‑Hub 系统与硬件设备及外部环境的接口规范，明确交互的数据格式、控制逻辑及通信协议，确保软硬件之间的稳定、高效交互。

### 3.2.1 硬件类型与支持范围  
-`支持类型`：标准 x86 架构服务器、台式计算机、笔记本电脑。  
-`最低配置要求`：  
CPU：Intel Core i3 或同等性能处理器； 内存：4GB RAM； 存储：10GB 可用硬盘空间（用于系统文件、SQLite 数据库）； 网络：稳定的有线 / 无线网络连接（10Mbps 以上）。  
-`推荐配置`：CPU：Intel Core i5 及以上； 内存：8GB RAM 及以上； 存储：20GB 及以上。  
### 3.2.2 客户端硬件  
-`支持类型`：个人计算机（PC）、笔记本电脑、平板电脑、智能手机。  
-`核心硬件要求`：处理器：支持现代浏览器的 ARM 或 x86 架构处理器； 内存：2GB RAM 及以上； 显示设备：分辨率不低于 1366×768（PC），支持触控操作（平板 / 手机）； 输入设备：键盘、鼠标（PC）、触摸屏（移动设备）。  
### 3.2.3 软件与本地存储（SQLite）交互  
-`数据读取`：后端软件（FastAPI）启动时，读取 SQLite 数据库中存储的菜单信息、订单记录、管理员账户信息。  
-`数据写入`：用户创建订单、管理员修改菜单 / 订单时，后端软件将数据写入 SQLite 数据库。  
-`控制信息`：数据库连接状态校验、事务提交 / 回滚控制、数据读写权限控制。  
### 3.2.4 通信协议与标准  
-`HTTP/HTTPS 协议`：客户端与服务器之间的所有数据交互均基于 HTTP/1.1 或 HTTP/2 协议进行。  
-`RESTful API 规范`：后端接口采用 RESTful 架构风格，通过标准的 HTTP 方法（GET、POST、PUT、DELETE）实现对资源的操作。  
### 3.2.5 SQLite 数据库接口  
-后端软件（Python）通过 Python 标准库 sqlite3 与本地 SQLite 数据库文件（.db）进行直接交互。  
-采用文件级锁机制，确保多用户并发访问时的数据一致性，避免数据冲突。  
### 3.2.6 接口异常与容错机制  
为确保系统稳定性，软硬件接口需具备以下容错能力：  
-`网络异常处理`：客户端与服务器通信中断时，展示友好的错误提示（如 “网络连接异常，请检查网络后重试”），并缓存用户当前操作数据（如购物车内容），待网络恢复后自动提交。  
-`硬件故障处理`：服务器存储硬件故障（如硬盘损坏）导致数据库无法读写时，系统自动宕机保护，避免数据丢失，并记录错误日志；  
-`协议兼容`：确保在老旧浏览器（如 IE10+，虽不推荐但需兼容）上核心功能可用，不依赖前沿 HTML5/CSS3 特性；对不支持 HTTPS 的老旧环境，降级支持 HTTP 访问，但强制提示安全风险。

-必须使用 JSON 作为前后端唯一数据交换格式。

-不允许多进程直接访问同一块内存数据。  
-数据库为单文件形式，不支持分布式部署。  
-所有组件必须运行在同一台设备上。

## 3.4 通信接口

本节定义 OrderSphere‑Hub 系统所使用的网络通信、协议、消息格式、安全加密、传输速率与同步机制等需求，覆盖所有与外部及内部通信相关的约束。  
### 3.4.1 支持的通信功能与标准  
系统依赖以下通信功能与标准：  
-`网络通信`：TCP/IP 协议簇  
-`Web 访问`：HTTP/1.1、HTTP/2、HTTPS  
-`数据交换`：JSON 格式  
-`浏览器通信`：现代浏览器标准 Ajax / Fetch API  
-`无外部邮件、电子表格、第三方消息队列直接集成`  
### 3.4.2 通信协议与消息格式  
-前端 ↔ 后端：HTTP/HTTPS  
-接口风格：RESTful API  
-请求方法：GET、POST、PUT、DELETE  
### 3.4.3 通信安全与加密需求  
-生产环境必须使用 HTTPS 加密传输  
-管理员登录使用Token 简单鉴权，无复杂 OAuth  
-密码不在 URL 中传递，仅在 POST body 传输  
-禁止明文存储敏感信息  
-接口访问需来源合法，支持基础跨域限制（CORS）  
### 3.4.4 数据传输速率需求  
-单接口响应时间：≤ 1 秒  
-菜单列表数据量：≤ 500KB  
-订单数据量：≤ 10KB / 条  
-支持最低网络带宽：≥ 1Mbps  
### 3.4.5 同步通信机制  
-系统采用同步请求‑响应模型：前端请求 → 后端处理 → 立即返回结果  
-无异步消息队列  
-无长连接、无 WebSocket、无推送  
-数据一致性：请求完成即数据落地  
-订单提交、购物车修改均为同步确认

## 4.3 功能需求

### 4.3.1 菜单展示与查询
FR-4.3-001: 展示全部菜品列表
用户场景：普通点餐用户、临时访客、管理员。
功能描述：
输入：用户打开系统首页或点击“菜单”导航入口。
处理逻辑：
1.  系统向后端发送菜品列表查询请求；
2.  后端从SQLite数据库读取所有有效菜品数据（状态为“可用”）；
3.  按菜品分类排序，整理数据为JSON格式返回前端；
4.  前端接收数据后，以卡片形式渲染菜品列表（包含名称、价格、描述）。
输出：
成功：页面展示所有可用菜品列表，按分类分区显示。
失败：显示“菜单加载失败，请重试”提示，提供“重试”按钮。
错误处理：
- 数据库连接异常：提示“系统暂时无法加载菜单，请联系管理员”；
- 无有效菜品：显示“暂无菜品可供选择”，无其他操作按钮。

FR-4.3-002: 按分类筛选菜品
用户场景：普通点餐用户、临时访客、管理员。
功能描述：
输入：用户点击页面分类标签（如“热菜”“凉菜”“主食”）。
处理逻辑：
1.  系统获取用户选择的分类标识；
2.  后端根据分类标识筛选对应菜品数据；
3.  前端接收筛选后的数据，重新渲染菜品列表。
输出：
成功：页面展示该分类下的所有菜品，分类标签高亮显示，同步更新筛选结果数量。
失败：显示“该分类暂无菜品”提示。
错误处理：
- 分类标识非法（不存在的分类）：提示“筛选失败，分类不存在”，自动恢复显示全部菜品。

FR-4.3-003: 按关键词搜索菜品
用户场景：普通点餐用户、临时访客、管理员。
功能描述：
输入：用户在搜索框输入菜品名称关键词，点击“搜索”按钮或按下Enter键。
处理逻辑：
1.  系统获取用户输入的关键词，去除首尾空格；
2.  后端按关键词模糊匹配菜品名称（不区分大小写）；
3.  筛选出匹配的菜品数据，返回前端并渲染。
输出：
成功：页面展示匹配的菜品列表，显示“找到X条结果”提示。
失败：显示“未找到匹配菜品，请更换关键词”提示。
错误处理：
- 关键词为空/仅含空格：提示“请输入搜索关键词”，不执行搜索操作；
- 搜索超时：提示“搜索超时，请重试”，提供“重试”按钮。

### 4.3.2 购物车管理
FR-4.3-004: 添加菜品到购物车
用户场景：普通点餐用户。
功能描述：
输入：用户点击菜品卡片的“加入购物车”按钮，系统自动获取当前菜品ID、名称、价格，默认数量为1（可手动修改后添加）。
处理逻辑：
1.  检查当前菜品是否已在购物车中；
2.  若已存在：更新该菜品的数量（原数量+新增数量），实时计算小计金额；
3.  若不存在：新增购物车条目，记录菜品ID、名称、价格、数量及小计金额；
4.  实时计算购物车总金额。
输出：
成功：显示“菜品已添加至购物车”弹窗提示，购物车图标显示当前商品数量，同步更新购物车列表及总金额。
失败：显示“添加失败，请重试”提示。
错误处理：
- 菜品已下架/不可用：提示“该菜品已下架，无法添加”；
- 数量输入非法（非数字/负数）：自动修正为1，添加后提示“数量已修正为1”。

FR-4.3-005: 修改购物车菜品数量
用户场景：普通点餐用户。
功能描述：
输入：用户在购物车列表中，点击数量“+”“-”按钮，或直接输入数量并提交。
处理逻辑：
1.  校验输入的数量是否为正整数（≥1）；
2.  若数量合法：更新该菜品的数量，重新计算小计金额及购物车总金额；
3.  若数量为1时点击“-”：提示用户是否删除该菜品。
输出：
成功：实时更新购物车中该菜品的数量、小计金额及总金额，无额外提示（点击“-”至1时除外）。
失败：显示“请输入有效数量（≥1）”提示，保持原数量不变。
错误处理：
- 数量为0/负数/非数字：自动恢复为原数量，提示“请输入1以上的整数”；
- 数量输入过长（超过99）：自动限制为99，提示“单次最多添加99份”。

FR-4.3-006: 删除购物车单个菜品
用户场景：普通点餐用户。
功能描述：
输入：用户点击购物车中某菜品的“删除”按钮，确认删除操作。
处理逻辑：
1.  弹出确认弹窗，询问用户“是否确认删除该菜品？”；
2.  用户点击“确认”：从购物车中移除该菜品条目，重新计算购物车总金额；
3.  用户点击“取消”：终止删除操作，保持购物车不变。
输出：
成功：显示“菜品已从购物车移除”提示，购物车列表刷新，总金额同步更新；若购物车为空，隐藏“提交订单”按钮。
失败：无提示，保持购物车原状态（用户取消操作时）。
错误处理：
- 菜品已不在购物车中：提示“操作无效，该菜品不在购物车中”。

FR-4.3-007: 清空购物车
用户场景：普通点餐用户。
功能描述：
输入：用户点击购物车页面的“清空购物车”按钮，确认清空操作。
处理逻辑：
1.  弹出确认弹窗，询问用户“是否确认清空购物车？”；
2.  用户点击“确认”：删除购物车中所有菜品条目，重置总金额为0；
3.  用户点击“取消”：终止清空操作，保持购物车不变。
输出：
成功：显示“购物车已清空”提示，购物车列表显示空状态，隐藏“提交订单”按钮。
失败：无提示，保持购物车原状态（用户取消操作时）。
错误处理：
- 购物车已为空：提示“购物车已为空，无需清空”。

### 4.3.3 订单创建与查询
FR-4.3-008: 提交订单
用户场景：普通点餐用户。
功能描述：
输入：用户点击购物车的“提交订单”按钮，确认订单信息（菜品列表、总金额）。
处理逻辑：
1.  校验购物车是否为空，若为空则终止操作；
2.  系统生成唯一订单号（格式：年月日+随机6位数字）；
3.  将订单信息（订单号、菜品列表、总金额、创建时间、订单状态）写入SQLite数据库；
4.  提交成功后，清空当前购物车。
输出：
成功：显示“订单提交成功”提示，跳转至订单详情页，展示订单号、菜品列表、总金额、创建时间及“待确认”状态。
失败：显示“订单提交失败，请重试”提示，保留购物车内容。
错误处理：
- 购物车为空：提示“购物车为空，无法提交订单”，禁用“提交订单”按钮；
- 数据库写入失败：提示“订单提交失败，请联系管理员”，保留购物车内容；
- 网络中断：提示“网络连接异常，订单提交失败，请检查网络后重试”。

FR-4.3-009: 查询个人订单列表
用户场景：普通点餐用户。
功能描述：
输入：用户点击导航栏“我的订单”入口，请求查看个人订单。
处理逻辑：
1.  系统读取当前用户的所有订单数据（按创建时间倒序排列）；
2.  筛选订单状态（待确认、已完成、已取消），整理为JSON格式返回前端；
3.  前端渲染订单列表，显示订单号、总金额、创建时间、订单状态。
输出：
成功：页面展示个人所有订单列表，按时间倒序排列，可按状态筛选。
失败：显示“订单加载失败，请重试”提示，提供“重试”按钮。
错误处理：
- 无个人订单：显示“暂无订单，快去点餐吧”提示；
- 网络异常：提示“网络连接异常，无法加载订单”。

FR-4.3-010: 查看订单详情
用户场景：普通点餐用户。
功能描述：
输入：用户点击订单列表中的某条订单，请求查看详情。
处理逻辑：
1.  系统根据订单号，从数据库中读取该订单的详细信息（订单号、菜品列表、单价、数量、小计、总金额、创建时间、订单状态）；
2.  整理数据并返回前端，前端渲染订单详情页面。
输出：
成功：页面展示该订单的完整详情，包含所有菜品信息及订单基本信息。
失败：显示“订单详情加载失败，请重试”提示。
错误处理：
- 订单号非法/不存在：提示“订单不存在，请刷新页面后重试”。

### 4.3.4 管理员身份认证
FR-4.3-011: 管理员登录
用户场景：管理员（店长、后厨负责人等）。
功能描述：
输入：管理员在登录弹窗中，输入用户名（默认admin）、密码（默认admin123），点击“登录”按钮。
处理逻辑：
1.  校验用户名和密码是否为空；
2.  比对输入的用户名、密码与数据库中存储的管理员信息是否一致；
3.  若一致，生成临时权限令牌，维持登录状态（有效期2小时）；
4.  若不一致，记录错误登录次数，累计3次后，暂时禁止登录10分钟。
输出：
成功：显示“登录成功”提示，跳转至管理员操作界面（菜单管理/订单管理）。
失败：显示“账号或密码错误，请重新输入”提示，显示剩余可尝试次数。
错误处理：
- 用户名/密码为空：提示“用户名和密码不能为空”；
- 累计3次登录失败：提示“登录失败次数过多，请10分钟后再尝试”；
- 令牌失效：提示“登录状态已过期，请重新登录”。

FR-4.3-012: 管理员退出登录
用户场景：已登录的管理员。
功能描述：
输入：管理员点击页面右上角“退出登录”按钮。
处理逻辑：
1.  清除管理员的登录令牌，终止登录状态；
2.  跳转至系统首页，隐藏管理员操作入口。
输出：
成功：显示“已成功退出登录”提示，跳转至系统首页。
失败：无提示，强制跳转至系统首页，清除登录状态。
错误处理：
- 退出时网络异常：强制清除本地登录状态，提示“已退出登录，若页面未刷新，请手动刷新”。

FR-4.3-013: 管理员权限校验
用户场景：未登录用户、已登录管理员。
功能描述：
输入：未登录用户尝试访问管理员接口（菜单管理、订单管理）。
处理逻辑：
1.  系统校验当前用户是否有管理员登录令牌；
2.  若有令牌且有效：允许访问管理员接口；
3.  若无令牌/令牌失效：拒绝访问，强制跳转至登录弹窗。
输出：
成功：已登录管理员正常访问管理员操作界面。
失败：未登录用户/令牌失效时，弹出登录弹窗，提示“请先登录管理员账号”。
错误处理：
- 非法访问管理员接口：提示“无操作权限，请登录管理员账号”，拒绝加载接口数据。

输出：
成功：页面展示所有菜品的完整数据，包含操作按钮（修改、启用/禁用）。
失败：显示“菜单数据加载失败，请重试”提示。
错误处理：

FR-4.3-015: 管理员修改菜单信息
用户场景：已登录的管理员。
功能描述：
输入：管理员点击菜单列表中的“修改”按钮，输入修改后的菜品信息（名称、价格、描述、状态），点击“提交”按钮。

输出：

失败：显示“菜单修改失败，请重试”提示，保留原信息。

- 名称/价格为空：提示“菜品名称和价格不能为空”；

- 菜品ID非法：提示“菜品不存在，无法修改”。

FR-4.3-016: 管理员启用/禁用菜品
用户场景：已登录的管理员。
功能描述：
输入：管理员点击菜单列表中的“启用”/“禁用”按钮，切换菜品状态。
处理逻辑：
1.  系统获取当前菜品ID及当前状态；
2.  切换菜品状态（可用→不可用，不可用→可用），更新数据库中的菜品状态；
3.  同步更新前端菜单列表的菜品状态及操作按钮。
输出：
成功：显示“菜品状态已更新”提示，菜单列表中该菜品状态同步切换。
失败：显示“状态更新失败，请重试”提示，保持原状态不变。
错误处理：
- 菜品ID非法：提示“菜品不存在，无法更新状态”。

### 4.3.6 订单数据管理
FR-4.3-017: 管理员查询全部订单
用户场景：已登录的管理员。
功能描述：
输入：管理员点击“订单管理”入口，请求查看所有用户的订单数据。
处理逻辑：
1.  系统从SQLite数据库读取所有订单数据（包含所有用户、所有状态）；
2.  按订单创建时间倒序排列，整理数据（订单号、用户信息、菜品列表、总金额、创建时间、订单状态）并返回前端；
3.  前端渲染订单管理列表，支持按订单号、订单状态筛选。
输出：

### 4.3.9 异常处理与提示
FR-4.3-023: 网络异常处理
用户场景：所有用户（普通用户、管理员、临时访客）。
功能描述：
输入：用户执行操作时（如加载菜单、提交订单、登录），出现网络中断、网络延迟。
处理逻辑：
1.  系统捕获网络异常信号，终止当前操作；
2.  生成友好的错误提示，提供“重试”按钮；
3.  缓存用户当前未完成的操作（如购物车内容、未提交的订单），待网络恢复后可继续操作。
输出：
成功：网络恢复后，用户点击“重试”，可继续完成之前的操作。
失败：网络持续异常时，提示“网络连接持续异常，请检查网络设置后重新访问”。
错误处理：
- 网络延迟超时：提示“网络响应超时，请重试”；
- 网络中断：提示“网络连接已中断，请检查网络后重试”。

FR-4.3-024: 非法请求处理
用户场景：所有用户、非法访问者。
功能描述：
输入：用户发送非法请求（如访问不存在的接口、修改请求参数、非法调用接口）。
处理逻辑：
1.  系统校验请求的合法性（接口地址、请求参数、权限）；
2.  若为非法请求：拒绝处理，记录错误日志（请求地址、时间、IP）；
3.  生成统一的错误提示，不暴露技术细节。
输出：
成功：拒绝非法请求，返回统一错误提示。
失败：无额外失败场景，确保所有非法请求均被拦截。
错误处理：
- 访问不存在的接口：提示“请求的页面不存在，请检查地址后重试”；
- 修改请求参数：提示“请求参数非法，操作失败”；
- 非法调用管理员接口：提示“无操作权限，请登录管理员账号”。

FR-4.3-025: 数据校验异常处理
用户场景：所有用户。
功能描述：
输入：用户输入非法数据（如非数字价格、空字段、超长文本），提交操作。
处理逻辑：
1.  系统对输入数据进行校验，检查字段格式、长度、合法性；
2.  若数据非法：终止操作，生成针对性的错误提示；
3.  保留用户已输入的合法数据，方便用户修正。
输出：
成功：用户修正数据后，可正常提交操作。
失败：显示针对性错误提示，不执行操作。
错误处理：
- 空必填字段：提示“请填写必填字段（如菜品名称、价格）”；
- 数据格式非法：提示“请输入合法格式的数据（如价格为数字）”；
- 文本超长：提示“输入内容过长，请控制在X字符以内”。

  7a2. 购物车数据保持不变，用例结束

## 5.1 性能需求

本章阐述OrderSphere-Hub在线餐厅点餐系统在不同应用场景下的性能需求，明确各功能模块的性能指标、原理及验证标准，为开发人员提供设计依据，确保系统在预期负载下高效、稳定运行。性能需求按核心功能模块分类，明确用户并发量、响应时间、容量限制等关键指标，同时解释指标设定原理，助力开发人员做出合理设计选择。
1.并发性能需求

原理：餐饮场景存在用餐高峰（如午餐11:30-13:30、晚餐17:30-19:30），此时用户并发操作集中，需保障系统在高并发下无卡顿、无崩溃，核心操作正常响应，避免因并发量不足影响餐厅正常运营。

- 普通用户并发：在餐厅用餐高峰时段（11:30-13:30、17:30-19:30），支持同时在线普通用户≥50人，并发下单请求≥10次/分钟，并发浏览菜品请求≥30次/分钟，系统无卡顿、无响应延迟。

- 管理员并发：支持同时在线管理员≤5人，并发操作（菜品上下架、订单处理、库存调整）≥3次/分钟，多管理员同时操作无冲突，操作结果实时同步，无数据错乱。

- 支付并发：高峰时段支持并发支付请求≥8次/分钟，支付操作与订单提交操作互不干扰，支付请求响应独立，不影响其他操作（如浏览菜品、购物车修改）。

- 验证标准：在4核8G内存服务器、Windows 10操作系统环境下，通过压力测试模拟高峰并发场景，持续30分钟，系统无崩溃、无数据丢失，核心操作响应正常。

2.响应时间需求

原理：响应时间直接影响用户体验，普通用户对操作延迟敏感（如下单、支付），管理员操作（如订单处理、数据统计）需高效完成，避免影响餐厅运营效率，因此需按功能模块明确响应时间标准。

- 普通用户核心操作：在网络带宽≥10Mbps、终端设备（手机/电脑）配置≥4G内存的环境下，95%的操作响应时间需满足：菜品浏览、搜索响应≤0.5秒；购物车添加、修改、删除响应≤1秒；订单提交响应≤1.5秒；支付请求响应≤2秒（不含第三方支付平台跳转时间）；订单查询、个人信息修改响应≤1秒。

- 管理员核心操作：在服务器配置（4核8G内存、SQLite数据库）环境下，95%的操作响应时间需满足：菜品新增、编辑、上下架响应≤1.5秒；订单查看、接单、退款响应≤2秒；用户管理（新增、删除、密码重置）响应≤1秒；数据备份响应≤30秒（单数据库备份，数据量≤100MB）；日志查询响应≤1.5秒。

- 系统全局响应：在非高峰时段，所有用户操作的平均响应时间≤1秒，峰值响应时间≤3秒；高峰时段（并发量达上限），核心操作（下单、支付、接单）响应时间≤3秒，无超时失败情况。

- 异常响应：当操作响应时间超出上述标准时，系统需在1秒内弹出“操作繁忙，请稍后再试”的提示，并记录响应超时日志，便于开发人员排查优化。

3.容量需求

原理：系统长期运行会产生大量用户数据、订单数据、日志数据，需明确存储容量、数据行数限制，确保数据库、服务器存储可满足长期使用需求，避免因容量不足导致数据丢失或系统卡顿。

- 数据库容量：SQLite数据库支持最大存储容量≥10GB，满足系统连续运行1年（按每日100笔订单、50名用户操作计算）的数据存储需求；支持数据库表最大行数：用户表≤1000行（适配中小型餐厅用户规模），订单表≤100000行，菜品表≤500行，日志表≤50000行。

- 服务器存储：系统部署服务器需预留≥20GB空闲磁盘空间，用于安装系统、存储备份数据及日志文件；其中备份数据单独占用≥5GB空间，支持每日自动备份，备份文件保留近30天。

- 终端存储：普通用户使用的终端设备（手机、电脑）需预留≥100MB存储空间，用于存储系统缓存、本地操作记录，确保前端页面加载流畅，无存储不足提示。

- 数据留存容量：订单数据、用户操作日志留存≥6个月，留存数据占用存储空间≤5GB；过期数据可手动归档，归档文件占用存储空间≤10GB，归档后不影响系统正常运行。

4.稳定性性能需求

原理：系统稳定性直接影响餐厅运营，需保障长期运行无故障、无数据错乱，尤其是高峰时段，避免因系统故障导致订单丢失、支付异常等损失，因此明确稳定性指标及故障恢复要求。

- 系统连续运行：在正常服务器环境（4核8G内存、稳定网络）下，系统可连续无故障运行≥72小时，无卡顿、无崩溃；高峰时段（每日2小时高峰）连续运行≥30天，无核心功能故障。

- 故障恢复：系统出现普通故障（如服务卡顿、操作超时）时，自动恢复时间≤5分钟；出现严重故障（如数据库异常、服务崩溃）时，手动恢复时间≤30分钟，恢复后数据无丢失、无错乱。

- 数据一致性：并发操作时，数据同步延迟≤1秒，确保多用户同时操作同一数据（如同一菜品库存）时，数据无错乱、无重复（如重复下单导致库存异常）。

- 网络适应性：在网络带宽≥10Mbps时，系统正常运行；网络带宽降至5Mbps时，核心操作（下单、支付、接单）仍可正常响应，仅非核心操作（如日志查询）可略有延迟。

5.各功能模块专项性能需求

针对系统核心功能模块，单独明确专项性能需求，确保各模块性能达标，衔接整体系统性能。

- 菜品管理模块：批量导入菜品（≤50条/次）响应时间≤5秒，导入过程中不影响其他操作；菜品图片加载（单张图片≤2MB）响应≤0.8秒，批量加载（≤10张）响应≤3秒。

- 订单管理模块：订单查询（按时间、状态筛选）响应≤1秒，批量导出订单（≤1000条/次）响应≤10秒；订单状态修改（接单、取消、退款）响应≤1.5秒，状态实时同步至前端。

- 用户管理模块：批量创建普通用户（≤20个/次）响应≤3秒；用户密码重置、账号锁定/解锁响应≤1秒，操作结果实时反馈。

- 支付模块：支付请求提交后，与第三方支付渠道对接响应≤2秒；支付结果同步至系统≤1秒，支付失败提示响应≤0.5秒，同时记录失败原因。

- 日志管理模块：日志记录响应≤0.5秒，日志查询（按关键词、时间筛选）响应≤1.5秒；日志导出（≤1000条）响应≤5秒。

## 5.2 安全设施需求

本章详尽陈述OrderSphere-Hub在线餐厅点餐系统使用过程中，应对可能发生的损失、破坏或危害所需的安全保护动作、预防措施，明确系统必须采取的安全设施，界定需遵从的安全标准、策略及规则，确保系统在各类风险场景下可有效规避或降低危害，所有需求均具体、可验证。
1.操作风险安全设施需求

针对系统操作过程中可能导致数据丢失、操作异常、财产损失的风险，明确需采取的安全保护动作及预防措施：

- 当系统检测到单笔订单支付金额异常（超出菜品定价总和±10%）时，必须立即终止支付操作，1秒内弹出“支付金额异常，请核对菜品价格后重新提交”的提示，并记录该异常操作（操作人、操作时间、异常金额），同步通知管理员核查。

- 当用户（管理员/普通用户）连续5次尝试操作（下单、退款、数据修改）失败时，系统必须暂停该用户对应操作权限15分钟，同时记录操作日志（含IP地址、操作内容），防止恶意操作导致系统故障或数据破坏。

- 当检测到非授权IP尝试访问管理员后台时，必须立即拦截访问请求，记录IP地址、访问时间及访问行为，连续3次拦截后，自动将该IP加入临时黑名单（24小时），禁止其访问系统任何模块。

- 当系统执行批量操作（批量删除用户、批量修改菜品信息）时，必须先自动备份对应数据（备份完成后给出明确提示），若操作过程中出现中断（如网络中断、系统卡顿），立即终止操作并自动恢复数据，避免批量数据丢失或错乱。

- 当普通用户尝试取消已支付订单时，系统必须先验证订单状态（是否已接单、是否已备餐），若订单已处于备餐状态，立即终止取消操作，提示“订单已备餐，无法取消，请联系管理员申请退款”，同时记录该取消尝试。

2.数据安全设施需求

针对数据泄露、数据篡改、数据丢失等危害，明确系统必须采取的安全保护动作及预防措施：

- 当系统检测到敏感数据（用户手机号、订单详情）被异常读取（如非授权账号尝试访问）时，必须立即终止数据读取操作，加密隐藏敏感数据，同时记录异常行为，触发管理员告警（弹窗+日志提醒）。

- 当数据库存储容量达到90%时，系统必须自动触发容量预警，提示管理员清理过期数据（如6个月前的日志），同时自动压缩非核心数据，防止数据库过载导致数据丢失或系统崩溃。

- 当检测到数据修改行为（如修改菜品价格、订单金额）与历史操作习惯不符（如管理员首次修改菜品价格且幅度超过50%）时，必须终止修改操作，要求管理员二次输入密码确认，同时记录该异常修改尝试。

- 系统备份数据在存储过程中，必须采用双重加密（文件加密+存储路径加密），若检测到备份数据被非法篡改，立即锁定备份文件，自动启用上一次有效备份，同时通知超级管理员核查。

- 当用户删除自身账号时，系统必须先验证用户身份（密码+手机验证码），确认后仅删除账号关联信息，保留订单、操作日志等核心数据（不可删除），防止用户误删导致业务数据丢失。

3.系统运行安全设施需求

针对系统运行过程中可能出现的崩溃、卡顿、恶意攻击等危害，明确系统必须采取的安全保护动作及预防措施：

- 当系统CPU使用率超过85%、内存占用超过80%时，必须自动限制非核心操作（如批量导出数据、非紧急下单），优先保障核心操作（支付、订单提交）正常运行，同时弹出“系统负载过高，请稍后操作”的提示，记录系统负载数据。

- 当检测到恶意代码注入、SQL注入等攻击行为时，必须立即拦截攻击请求，关闭对应接口访问权限（10分钟内自动恢复），记录攻击细节（攻击方式、IP地址、攻击时间），同步触发管理员告警。

- 系统运行过程中，若出现服务中断（如后端服务崩溃），必须在5分钟内自动重启服务，重启后自动恢复未完成的核心操作（如未提交的订单、未完成的支付），避免数据丢失，同时记录故障详情便于排查。

- 当系统遭遇断电、突然关机等意外情况时，必须在重启后自动恢复所有未保存的操作数据（如未提交的购物车、未完成的订单），并弹出提示“系统已恢复，可继续操作”，防止用户重复操作导致损失。

- 系统必须定期（每日凌晨2点）自动检测核心模块（支付模块、订单模块、数据存储模块）的运行状态，若检测到模块异常（如无法正常响应），立即启动备用模块，确保业务正常开展，同时通知管理员排查原模块故障。

4.合规性安全设施需求

明确系统必须遵从的安全标准、策略，界定对应的安全保护动作，确保符合相关规范：

- 系统必须遵从《个人信息保护法》相关要求，对用户个人信息（手机号、邮箱）进行加密隐藏展示（如手机号显示为138****1234），禁止明文展示，若检测到明文展示敏感信息，立即隐藏并提示管理员整改。

- 系统所有安全设施必须符合餐饮行业数据安全规范，针对订单支付、用户信息存储等操作，必须留存完整操作日志，日志留存时间不少于6个月，若检测到日志缺失或篡改，立即锁定系统相关操作权限，通知管理员核查。

- 系统支付模块必须对接合规支付渠道，若检测到非合规支付渠道接入，立即终止支付功能，禁止任何支付操作，同时记录接入信息，通知管理员处理。

- 系统必须建立安全应急机制，当发生重大安全事件（如大规模数据泄露、恶意攻击导致系统瘫痪）时，必须立即终止所有操作，启动应急备份系统，同时通知超级管理员及相关负责人，2小时内给出应急处理方案。

5.安全设施遵从标准

- 系统所有安全设施必须严格遵从以下标准，确保安全保护动作有效、规范：

- 安全设施需与系统版本同步更新，每次系统迭代后，需检测安全设施的兼容性和有效性，确保可正常应对各类风险。

- 所有安全保护动作需可验证、可追溯，操作日志需完整记录安全设施的触发时间、触发原因、执行结果，便于后续审计。

## 5.3 安全性需求

本章详尽陈述OrderSphere-Hub在线餐厅点餐系统的安全性需求，涵盖用户身份确认、操作授权、数据保护、保密策略等核心内容，保障系统使用安全、数据隐私安全，明确系统需满足的安全约束，所有需求均为可验证、可落地的具体要求。
1.用户身份确认需求

- 用户注册时，需完成身份基础校验，普通用户需填写有效手机号（接收验证码完成验证），管理员需由超级管理员手动创建并分配初始账号密码，禁止匿名注册。

- 所有用户登录时，需输入账号+密码完成身份验证，密码需区分大小写，输入错误时给出“账号或密码错误”的模糊提示，不明确告知具体错误项（避免暴力破解）。

- 用户首次登录系统（含管理员初始登录），必须更改系统分配的初始密码，初始密码不可重用，新密码需符合密码复杂度要求（8-20位，包含大小写字母、数字及1个特殊符号）。

- 支持密码找回功能，普通用户通过绑定手机号接收验证码找回，管理员通过超级管理员授权重置密码，密码重置后需强制重新登录并修改为新密码。

- 系统支持账号绑定功能，普通用户可绑定手机号、邮箱，管理员仅可绑定工作邮箱，绑定后可通过绑定方式辅助登录，未绑定账号仅可通过账号密码登录。

2.操作授权需求

- 系统采用角色分级授权机制，分为超级管理员、普通管理员、普通用户三个角色，不同角色拥有对应操作权限，无权限操作时提示“无操作权限，请联系管理员”。

- 超级管理员拥有系统全部操作权限，可创建、修改、删除普通管理员账号，分配普通管理员操作权限，查看所有操作日志及敏感数据。

- 普通管理员仅拥有菜品管理、订单管理、普通用户管理、数据备份（不可恢复）、日志查看权限，无系统配置、管理员账号管理、数据恢复权限。

- 普通用户仅拥有自身账号管理、菜品浏览、购物车操作、订单提交与查询、个人信息修改权限，无权访问任何管理后台及其他用户数据。

- 敏感操作（如退款、密码修改、数据删除、菜品上下架）需二次验证，管理员操作需输入当前密码，普通用户操作需输入登录密码或手机验证码。

- 批量操作（批量删除用户、批量修改菜品信息）仅超级管理员可执行，普通管理员仅可单个操作，且操作后需记录操作日志，便于追溯。

3.数据安全与保护需求

- 所有用户敏感数据（密码、手机号、邮箱、订单详情）需采用加密存储，密码采用不可逆加密算法（如SHA-256），敏感数据传输过程中采用HTTPS加密，防止数据泄露。

- 用户个人信息（手机号、邮箱）仅本人可查看、修改，管理员仅可查看普通用户账号信息，不可查看其密码、手机号等敏感隐私数据。

- 订单数据、用户操作日志、系统运行日志需长期留存（不少于6个月），留存期间不可随意删除，仅超级管理员可导出、清理过期日志，清理前需进行数据备份。

- 系统定期自动备份数据（每日凌晨1点），备份数据采用加密存储，存储路径仅超级管理员可访问，备份数据保留近30天，支持手动备份及数据恢复。

- 禁止系统内存储无关敏感数据（如身份证号、银行卡完整信息），支付相关数据仅留存必要订单关联信息，不留存支付密码、银行卡密码等核心敏感信息。

- 数据修改、删除需留下操作痕迹，记录操作人、操作时间、操作内容，已删除数据需进入回收站（留存7天），7天后自动永久删除，超级管理员可手动恢复回收站数据。

4.保密性策略需求

- 系统需严格遵循“最小权限”原则，用户仅可访问自身角色权限范围内的数据，不可越权访问其他用户或管理员的敏感数据。

- 管理员操作日志、敏感数据操作记录（如退款、数据修改）仅超级管理员可查看，普通管理员仅可查看自身操作日志。

- 系统报错信息需模糊处理，避免泄露系统架构、数据库信息、敏感数据，如数据库连接失败时，提示“系统异常，请联系管理员”，不显示具体报错代码及数据库地址。

- 系统安装包、配置文件、备份数据需设置访问权限，仅授权用户可访问，禁止匿名访问，防止敏感配置及数据泄露。

- 用户退出登录后，系统自动清除当前会话信息、缓存数据，不可保留登录状态，再次访问需重新登录验证身份。

5.安全审计需求

- 系统自动记录所有用户的操作行为，包括登录、退出、操作内容、操作时间、IP地址、操作结果，形成完整操作日志，日志不可篡改、不可删除。

- 对敏感操作（密码修改、退款、数据删除、权限分配）进行重点审计，记录操作全流程，一旦出现异常操作，可通过日志追溯操作人及操作细节。

- 超级管理员可定期导出安全审计日志，对异常操作（如多次登录失败、越权操作尝试）进行排查，系统支持异常操作自动告警（弹窗提示管理员）。

## 5.5 业务规则

本章列举OrderSphere-Hub在线餐厅点餐系统的所有核心业务操作规则，明确不同用户在特定场景下可执行的操作，这些规则虽非直接功能需求，但为功能实现提供约束和依据，确保系统操作规范、符合餐饮业务逻辑。
1.管理员操作规则

- 只有持有管理员账号密码并成功登录系统后台的用户（系统管理员、餐厅管理人员），可执行所有管理员操作，包括用户管理、菜品管理、订单管理、系统配置等，普通用户无权访问管理员后台。

- 管理员仅可对普通用户进行新增、删除、密码重置操作，不可修改自身账号密码以外的管理员账号信息；密码重置后，需通知对应普通用户及时修改初始密码。

- 仅管理员可进行菜品上下架操作，菜品下架后，普通用户无法在前端浏览、添加该菜品至购物车；菜品库存调整需由管理员操作，调整后系统自动同步前端库存显示，库存为0时自动下架。

- 只有管理员可执行订单取消、退款操作，其中单笔订单退款金额超过500元时，需二次确认（输入管理员密码）后方可执行，避免误操作。

- 管理员可查看所有用户的操作日志、订单记录，但不可修改已完成订单的核心信息（如订单金额、菜品明细），仅可补充订单备注信息。

- 系统数据备份、恢复、缓存清理及语言配置等系统管理操作，仅管理员可执行，操作后系统自动记录操作日志，便于追溯。

2.普通用户（食客）操作规则

- 普通用户需完成注册并登录系统后，方可进行购物车操作、订单提交与支付，未登录用户仅可浏览菜品详情，无法执行下单相关操作。

- 普通用户仅可查看、操作自身的订单（查看订单详情、取消未支付订单），无权查看其他用户的订单信息；未支付订单可由用户自行取消，支付后需联系管理员申请取消及退款。

- 普通用户可修改自身个人信息（用户名、密码），但用户名修改后不可重复，密码修改需验证原密码，避免账号被盗。

- 普通用户添加菜品至购物车时，不可超过该菜品当前库存数量，超出库存时系统提示无法添加，需调整购买数量。

- 同一普通用户在同一时间段（30分钟内）不可重复提交相同菜品、相同数量的未支付订单，避免重复下单；未支付订单超时（默认30分钟）后，系统自动取消，释放对应菜品库存。

3.通用业务规则

- 所有用户（管理员、普通用户）登录系统后，连续30分钟无任何操作，系统自动退出登录，需重新登录方可继续操作，保障账号安全。

- 系统所有操作（登录、下单、取消订单、退款、菜品上下架等）均自动记录操作日志，包括操作人、操作时间、操作内容，日志不可删除，仅管理员可查看。

- 支付操作需在系统内完成，不可跳转至第三方非指定平台，支付成功后系统自动更新订单状态，支付失败时自动回滚订单数据，恢复菜品库存。

- 菜品价格、描述、分类等信息修改后，系统实时同步至前端，普通用户浏览时显示最新信息；修改记录自动留存，管理员可查看历史修改记录。

- 系统语言切换后，所有用户看到的界面语言同步更新，且用户下次登录时自动保留上次设置的语言偏好，无需重复切换。

4.《OrderSphere-Hub 常见问题与故障排查手册》

- 目标用户：所有用户（管理员、普通用户）、技术支持人员

- 核心内容：按用户类型分类，汇总系统使用过程中常见的操作问题、故障及解决方案；包括登录异常、操作失败、支付异常、系统卡顿、数据异常等场景；附故障排查流程、联系技术支持的方式及反馈渠道。

- 交付格式：PDF格式（可搜索）+ TXT格式（简洁版，便于快速查阅），同时将核心问题嵌入系统在线帮助中，支持关键词搜索。

- 交付标准：问题分类清晰（管理员类、普通用户类、通用类），每个问题明确“问题现象→可能原因→解决方案”，解决方案具体可操作；覆盖系统80%以上常见问题，定期可更新补充，标注文档版本及更新日期。

通用交付标准

所有随软件发行的用户文档，需统一遵循以下交付标准，确保一致性和易用性：

- 文档版本：所有文档版本需与系统版本一致，标注版本号（如V1.0）、编制日期、编制人，后续系统迭代时，同步更新对应文档并标注更新内容。

- 格式规范：PDF文档统一采用A4纸张大小，字体为宋体，标题加粗，正文字体大小11号，行间距1.5倍，段落清晰，目录可跳转；截图清晰、无模糊，标注明确，避免冗余信息。

- 语言要求：通俗易懂，避免专业术语堆砌，必要时对专业术语进行解释；语句简洁、逻辑清晰，步骤明确，确保不同操作水平的用户都能理解。

- 兼容性：PDF文档支持所有主流PDF阅读器（Adobe Acrobat、WPS、浏览器自带阅读器）打开；HTML格式在线帮助支持主流浏览器（Chrome、Edge、Firefox），适配不同屏幕尺寸。

- 完整性：文档内容需完整覆盖对应场景的所有核心操作和常见问题，无遗漏；操作步骤需连贯，确保用户按文档操作可顺利完成对应功能。

- 可维护性：可编辑版文档（Word、TXT）需结构清晰，便于后续系统迭代时更新；所有文档需留存源文件，便于修改和补充。

文档交付方式

所有用户文档随软件安装包一同交付，具体交付方式如下：

- 安装包内置：在系统安装包的“Documents”文件夹中，存放所有PDF、Word、TXT格式文档及相关脚本、模板文件，用户安装系统后可直接访问。

- 系统内置访问：在线HTML格式手册、在线图文教程、核心常见问题，直接嵌入系统前端和后端后台，用户可通过系统内“帮助”入口随时查看。

- 单独打包：提供单独的文档压缩包，便于用户单独下载、打印、分发（如餐厅张贴普通用户操作手册图文版）。
